The Story

You’ve been here before. Your API maps User to UserDto ten thousand times per second. Then one Friday, someone adds a Parent property that points back to itself. The stack overflows. Production goes down. Nobody knows why.

Or maybe it’s subtler. A mapping library that phones home with telemetry you never opted in to. Or one that only supports .NET 8+, leaving your legacy services stranded on .NET Framework without an upgrade path.

Mapture was built for teams who are tired of compromise. It is the only .NET mapper that is simultaneously:

  • Fastest — benchmarked faster than Mapster, AutoMapper, and PanoramicData.Mapper
  • Safest — cycle detection and max-depth enforcement catch infinite recursion before it happens
  • Broadest — multi-targets .NET Framework 4.8, .NET Standard 2.0, .NET 8, and .NET 10
  • Cleanest — zero telemetry, familiar API, 30-minute migration from AutoMapper
Bottom line: If you can write dest.Name = src.Name by hand, Mapture can do it for you at nearly the same speed — with cycle safety you’d never get writing it yourself.

Quick Start

Go from zero to mapping in four steps:

1

Install

dotnet add package Mapture
dotnet add package Mapture.Extensions.DependencyInjection
Supported frameworks: .NET Framework 4.8, .NET Standard 2.0, .NET 8, .NET 10
2

Define Your Models

public class User
{
    public int Id { get; set; }
    public string Name { get; set; }
    public string Email { get; set; }
}

public class UserDto
{
    public int Id { get; set; }
    public string Name { get; set; }
    public string Email { get; set; }
}
3

Create a Profile

using Mapture;

public class UserProfile : Profile
{
    public UserProfile()
    {
        CreateMap<User, UserDto>();
    }
}

Properties with matching names are mapped automatically. No boilerplate needed.

4

Map Objects

var config = new MapperConfiguration(cfg =>
{
    cfg.AddProfile<UserProfile>();
});

IMapper mapper = config.CreateMapper();

var user = new User { Id = 1, Name = "Alice", Email = "alice@example.com" };
var dto = mapper.Map<User, UserDto>(user);

Console.WriteLine(dto.Name); // "Alice"
Convention-based: Properties are matched by name automatically. No explicit mapping is needed for properties with matching names and compatible types.

How is Mapture this fast? At configuration time, Mapture compiles your mapping into a native delegate using System.Linq.Expressions. The compiled code is essentially dest.Name = src.Name; dest.Age = src.Age; — the same thing you’d write by hand. There’s no reflection on the hot path, no dictionary lookups per property, and no heap allocations beyond the destination object.

Why Not Zero Overhead?

The ~8 ns gap between Mapture and hand-written code comes from two things:

  • Delegate invocation — calling a compiled Func<T,R> costs ~2–3 ns more than inlined code
  • Cache lookup — finding the right delegate for the type pair adds ~3–5 ns (thread-static fast path)

For context, a single database query typically takes 500,000–5,000,000 ns. The 8 ns overhead is undetectable in any real application.

How It Works

Mapture’s speed comes from a three-layer architecture that does heavy work once (at configuration time) so that every subsequent mapping call is as cheap as possible.

Layer 1 — Configuration (runs once at startup)

When you call CreateMap<User, UserDto>(), Mapture builds a TypeMap that records which source property maps to which destination property, any custom resolvers, conditions, and before/after hooks. It also runs cycle analysis — a graph traversal that determines whether each type pair can ever reach itself through nested mappings.

Layer 2 — Compilation (runs once per type pair, on first use)

The first time you map User → UserDto, Mapture compiles an expression tree that does exactly what hand-written code would do: create a new UserDto, assign each property, and return it. For nested objects, the child’s compiled delegate is embedded directly into the parent’s expression tree as a constant — no lookup per call.

If the type pair is acyclic (the common case), the compiled delegate is pure property assignment. If the type pair is cyclic, Mapture wraps the delegate with a HashSet<object> visited tracker and a depth counter. You get safety where you need it and raw speed everywhere else.

Layer 3 — Execution (runs on every Map call)

The hot path is just three operations: null check → read cached delegate from a thread-static slot → call delegate. That’s it. No reflection, no dictionary lookups, no allocations beyond the destination object itself.

Think of it like a JIT compiler: the first call pays the compilation cost, but every subsequent call runs native code. Mapture does the same thing — except it compiles your mapping rules into delegates at runtime.

Profiles

Profiles organize your mappings into cohesive, reusable units. Each Profile is a class that groups related mappings together — typically one per domain area (Users, Orders, Products, etc.).

Create a Profile

using Mapture;

public class OrderProfile : Profile
{
    public OrderProfile()
    {
        CreateMap<Order, OrderDto>();
        CreateMap<OrderItem, OrderItemDto>();
    }
}

Register Profiles

You can register profiles one at a time or scan an entire assembly:

// Register individually
var config = new MapperConfiguration(cfg =>
{
    cfg.AddProfile<OrderProfile>();
    cfg.AddProfile<UserProfile>();
});

// Or scan an assembly for all Profile subclasses
var config = new MapperConfiguration(cfg =>
{
    cfg.AddProfiles(typeof(OrderProfile).Assembly);
});
Tip: Assembly scanning is the recommended approach for production applications. It automatically discovers all profiles, so you never forget to register a new one.

When to Use Profiles

  • One profile per bounded context — UserProfile, OrderProfile, InventoryProfile
  • Keep profiles small — 5–15 mappings per profile is a good target
  • Co-locate with models — put UserProfile next to User.cs and UserDto.cs

Fluent API

Mapture’s fluent API lets you customize every aspect of a mapping. All methods chain together and return the mapping expression, so you can configure everything in a single statement.

ForMember — Custom Property Mapping

Map a destination property from a custom expression:

CreateMap<User, UserDto>()
    .ForMember(d => d.FullName,
        opt => opt.MapFrom((Func<User, string>)(s => $"{s.First} {s.Last}")));

Ignore — Skip a Property

Tell Mapture not to map a destination property (also suppresses validation warnings):

CreateMap<User, UserDto>()
    .Ignore(d => d.InternalSecret);

ReverseMap — Bidirectional Mapping

Create the inverse mapping in one call:

CreateMap<User, UserDto>().ReverseMap();

// Now both directions work:
var dto  = mapper.Map<User, UserDto>(user);
var user = mapper.Map<UserDto, User>(dto);

ConvertUsing — Full Custom Conversion

Replace the entire mapping with a custom function. Useful for types that don’t follow conventions:

CreateMap<User, UserDto>()
    .ConvertUsing(src => new UserDto
    {
        Id   = src.Id,
        Name = src.Name.ToUpperInvariant()
    });
Note: When you use ConvertUsing, Mapture skips all convention-based mapping and calls your function directly. Make sure you map every property you need.

BeforeMap / AfterMap — Pre/Post Processing

Run code before or after the mapping executes:

CreateMap<User, UserDto>()
    .BeforeMap((src, dest) => Console.WriteLine($"Mapping user {src.Id}"))
    .AfterMap((src, dest) => dest.Name = dest.Name.Trim());

ConstructUsing — Custom Instantiation

Control how the destination object is created:

CreateMap<User, UserDto>()
    .ConstructUsing(src => new UserDto { Id = src.Id * 10 });

Condition — Conditional Mapping

Only map a property when a condition is true:

CreateMap<User, UserDto>()
    .ForMember(d => d.Age, opt =>
    {
        opt.MapFrom((Func<User, int>)(s => s.Age));
        opt.Condition(s => s.Age > 0);
    });

UseValue — Constant Value

Set a destination property to a fixed value regardless of the source:

CreateMap<User, UserDto>()
    .ForMember(d => d.Source, opt => opt.UseValue("API"));

Chaining Everything Together

All fluent methods return the mapping expression, so you can chain them:

CreateMap<User, UserDto>()
    .ForMember(d => d.FullName,
        opt => opt.MapFrom((Func<User, string>)(s => $"{s.First} {s.Last}")))
    .Ignore(d => d.InternalSecret)
    .AfterMap((src, dest) => dest.Name = dest.Name.Trim())
    .ReverseMap();

Safety Features

A mapping library that crashes production is worse than no mapping library at all. Mapture was designed with safety as a first-class concern — not an afterthought.

Cycle Detection

Objects that reference themselves (directly or through a chain) cause StackOverflowException in most mapping libraries. Mapture detects cycles at runtime and breaks them safely: 

var config = new MapperConfiguration(cfg =>
{
    cfg.CreateMap<Node, NodeDto>();
}, new MaptureOptions { EnableCycleDetection = true });

var node = new Node { Id = 1 };
node.Parent = node; // Circular reference!

var dto = mapper.Map<Node, NodeDto>(node);
// dto.Parent is null — cycle safely broken, no stack overflow
Smart cycle analysis: At configuration time, Mapture analyzes your type graph to find which type pairs can possibly have cycles. Types that cannot have cycles (the vast majority) get a zero-overhead fast path — no visited set, no depth counter, no per-call overhead. You get safety where you need it and raw speed everywhere else.

Max Depth Enforcement

Even without cycles, deeply nested object graphs can exhaust the stack. Mapture caps recursion at a configurable depth: 

new MaptureOptions { MaxDepth = 5 }
// Mapping stops at depth 5 and returns null for deeper nested objects

Configuration Validation

Catch unmapped properties at startup — not at 3am in production: 

var config = new MapperConfiguration(cfg =>
{
    cfg.CreateMap<User, UserDto>();
});

config.AssertConfigurationIsValid();
// Throws MaptureException if any destination property has no
// matching source property and isn't explicitly ignored

Dependency Injection

Mapture ships with first-class dependency injection support for ASP.NET Core and any IServiceCollection-based host.

Basic Registration

using Mapture.Extensions.DependencyInjection;

var builder = WebApplication.CreateBuilder(args);

// Scans the assembly for Profile classes, registers IMapper as singleton
builder.Services.AddMapture(typeof(Program).Assembly);

Registration with Options

builder.Services.AddMapture(typeof(Program).Assembly, options =>
{
    options.MaxDepth = 10;
    options.EnableCycleDetection = true;
    options.CompatibilityMode = false;
    options.EnableDebugTracing = false;
});

Using the Mapper

Inject IMapper into any controller, service, or handler:

public class UsersController : ControllerBase
{
    private readonly IMapper _mapper;

    public UsersController(IMapper mapper) => _mapper = mapper;

    [HttpGet]
    public IActionResult GetAll()
    {
        var users = _repository.GetAll();
        var dtos = users.Select(u => _mapper.Map<User, UserDto>(u)).ToList();
        return Ok(dtos);
    }
}
Lifetime: IMapper is registered as a singleton. The mapper is thread-safe and designed to be shared across the entire application. Do not create new mapper instances per request.

Migration from AutoMapper

Mapture was designed to be a near drop-in replacement for AutoMapper. The same Profile, CreateMap, ForMember, Ignore, and ReverseMap patterns all work identically. Most teams complete the migration in under 30 minutes.

Step-by-Step

1

Replace NuGet Packages

dotnet remove package AutoMapper
dotnet remove package AutoMapper.Extensions.Microsoft.DependencyInjection
dotnet add package Mapture
dotnet add package Mapture.Extensions.DependencyInjection
2

Find & Replace Namespaces

FindReplace
using AutoMapper;using Mapture;
using AutoMapper.Extensions.Microsoft.DependencyInjection; using Mapture.Extensions.DependencyInjection;
3

Replace DI Registration

// Before (AutoMapper)
services.AddAutoMapper(typeof(Startup));

// After (Mapture)
services.AddMapture(typeof(Startup));
4

Run Tests

Most mappings work identically. If you used AutoMapper-specific features that aren’t supported, enable compatibility mode: 

services.AddMapture(typeof(Startup), options =>
{
    options.CompatibilityMode = true;
});

What Stays the Same

  • Profile base class and CreateMap<S,D>()
  • ForMember(), Ignore(), ReverseMap()
  • ConvertUsing(), BeforeMap(), AfterMap()
  • ConstructUsing(), Condition()
  • IMapper interface with Map<S,D>(source)
  • Assembly scanning for automatic Profile discovery

What You Gain

  • Cycle detection and max-depth enforcement
  • .NET Framework 4.8 and .NET Standard 2.0 support

API Reference

IMapper

The primary interface for mapping objects. Obtained from MapperConfiguration.CreateMapper() or via dependency injection.

MethodDescription
Map<TDest>(object source) Maps source to TDest using runtime type resolution
Map<TSrc, TDest>(TSrc source) Maps source to a new TDest instance
Map<TSrc, TDest>(TSrc source, TDest dest) Maps source onto an existing destination object
Map(object source, Type srcType, Type destType) Non-generic mapping with explicit types

MapperConfiguration

Immutable configuration container. Build once at startup, then create mappers from it.

MethodDescription
CreateMapper() Creates an IMapper from this configuration
AssertConfigurationIsValid() Throws MaptureException if unmapped destination properties exist

Profile

Base class for grouping related mapping configurations:

MethodDescription
CreateMap<TSrc, TDest>() Define a mapping from TSrc to TDest. Returns IMappingExpression<TSrc, TDest>.

IMappingExpression<TSrc, TDest>

Fluent API for customizing a mapping. All methods return the expression for chaining.

MethodDescription
ForMember(dest, opt)Configure custom member mapping with MapFrom, Condition, or UseValue
Ignore(dest)Skip mapping for a destination member
ReverseMap()Create the inverse mapping TDest → TSrc
ConvertUsing(func)Replace entire mapping with a custom function
BeforeMap(action)Action to run before property mapping
AfterMap(action)Action to run after property mapping
ConstructUsing(func)Custom destination construction logic

MaptureOptions

Configuration options passed when creating a mapper:

PropertyTypeDefaultDescription
CompatibilityModeboolfalseEnable extra AutoMapper compatibility behaviors
MaxDepthint10Maximum recursion depth for nested mapping
EnableCycleDetectionbooltrueDetect and safely break circular references
EnableDebugTracingboolfalseEnable diagnostic debug logging

MaptureException

Thrown for configuration errors (unmapped properties) and mapping errors (unsupported types). Always includes a descriptive message.
